1. 系统架构

推荐采用分层架构：数据层负责空间数据存储与索引，服务层负责查询、切片和权限控制，前端层负责地图渲染、交互分析和业务展示。

数据源
Shapefile / GeoJSON / WFS / 遥感影像
处理层
ETL / 坐标转换 / 拓扑校验
服务层
PostGIS / GeoServer / Tile API
应用层
MapLibre / Dashboard / 业务系统

关键组件

组件 职责 建议选型
空间数据库 矢量数据存储、空间索引、拓扑关系计算 PostgreSQL + PostGIS
地图服务 WMS、WMTS、WFS、矢量切片发布 GeoServer 或 Tegola
前端地图 底图渲染、图层控制、空间交互 MapLibre GL JS
对象存储 影像、离线切片、导出文件存储 S3 兼容存储

2. 坐标与投影

GIS 系统必须统一坐标基准。业务数据入库时保留原始坐标信息，并在服务层输出统一坐标，避免前端临时转换导致误差和性能损耗。

  • Web 地图展示默认使用 EPSG:3857，适合互联网地图瓦片体系。
  • 经纬度数据交换默认使用 EPSG:4326，适合 GeoJSON、API 和跨系统共享。
  • 精密测绘、地籍、工程场景应保留本地投影坐标，并记录 EPSG 或自定义 WKT。
-- 将 WGS84 点数据转换到 Web Mercator
SELECT ST_Transform(geom, 3857) AS web_geom
FROM asset_points
WHERE ST_SRID(geom) = 4326;

3. 空间数据模型

空间表应将几何字段、业务字段、审计字段分离设计。几何字段统一命名为 geom，并为高频查询字段建立普通索引和空间索引。 

CREATE TABLE parcels (
  id BIGSERIAL PRIMARY KEY,
  parcel_code TEXT NOT NULL UNIQUE,
  owner_name TEXT,
  land_use TEXT NOT NULL,
  area_sqm NUMERIC(14, 2),
  geom GEOMETRY(MultiPolygon, 4326) NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX parcels_geom_gix ON parcels USING GIST (geom);
CREATE INDEX parcels_land_use_idx ON parcels (land_use);

几何类型约定

类型 用途 示例
Point 离散位置 摄像头、充电桩、门店
LineString 线性网络 道路、管线、河流
Polygon 区域边界 地块、行政区、服务范围

4. 数据入库流程

入库流程建议以批处理为主，实时数据通过消息队列增量写入。所有数据进入正式表前先进入暂存区，完成校验后再发布。

  1. 接收数据文件或接口数据，记录来源、时间、坐标系和负责人。
  2. 解析为标准空间格式，统一字段命名和编码。
  3. 执行坐标转换、几何修复、拓扑校验和重复数据检查。
  4. 写入暂存表，生成质量报告。
  5. 审核通过后合并到生产表，并刷新缓存和切片。
ogr2ogr \
  -f "PostgreSQL" PG:"host=db.example.com dbname=gis user=etl" \
  ./data/parcels.geojson \
  -nln staging.parcels_import \
  -lco GEOMETRY_NAME=geom \
  -nlt MULTIPOLYGON \
  -t_srs EPSG:4326

5. 地图切片服务

在线地图建议优先使用矢量切片。矢量切片体积小、样式灵活、适合多端展示。影像和复杂栅格数据使用栅格切片或云优化 GeoTIFF。

推荐策略： 热点区域预生成切片，低频图层按需生成并缓存；所有切片 URL 加版本号，便于灰度发布和缓存失效。 
GET /tiles/{layer}/{z}/{x}/{y}.pbf
Cache-Control: public, max-age=86400, stale-while-revalidate=604800

6. 空间查询 API

API 应以业务动作命名，并将空间过滤条件显式化。复杂空间计算放在数据库侧执行，前端只传递边界框、点位或缓冲距离。

查询附近资产

GET /api/assets/nearby?lng=121.4737&lat=31.2304&radius=500
SELECT id, name, type,
       ST_Distance(
         geom::geography,
         ST_SetSRID(ST_MakePoint($1, $2), 4326)::geography
       ) AS distance_m
FROM assets
WHERE ST_DWithin(
  geom::geography,
  ST_SetSRID(ST_MakePoint($1, $2), 4326)::geography,
  $3
)
ORDER BY distance_m
LIMIT 100;

7. 性能优化

  • 空间字段必须建立 GiST 或 SP-GiST 索引，高频范围查询先用边界框过滤。
  • 大面数据入库前做简化处理，按缩放级别输出不同精度。
  • 列表查询返回属性数据，地图查询返回 GeoJSON 或 MVT，避免一个接口承载所有需求。
  • 热点图层使用 CDN 缓存，API 查询使用短 TTL 缓存和用户权限维度隔离。
  • 超过 10 万要素的可视化应采用聚合、网格、热力或服务端切片。

8. 安全与权限

GIS 数据常包含敏感位置、资产和业务边界。权限模型需要同时覆盖接口访问、图层可见性、空间范围和导出能力。

控制点 要求
身份认证 统一接入 SSO、OAuth 或企业身份系统。
图层权限 按角色控制图层列表、字段可见性和样式配置。
空间权限 按行政区、项目范围或自定义多边形限制查询结果。
数据脱敏 敏感点位可做坐标偏移、聚合展示或最小缩放级别限制。

9. 部署与运维

前端文档和地图应用可部署到 Vercel，空间数据库、切片服务和 GeoServer 建议部署在受控云环境或内网 Kubernetes 集群。

发布检查清单

  • 确认生产环境 API 域名、地图 Token 和图层配置不写入源码。
  • 确认 PostGIS 扩展、空间索引和慢查询日志已启用。
  • 为切片、导出文件和影像配置对象存储生命周期策略。
  • 配置服务健康检查、错误告警、磁盘容量告警和数据库备份。
  • 将数据发布、样式发布和应用发布拆分，避免一次变更影响全部链路。
本页面部署方式： 当前技术文档是纯静态站点，可直接通过 Vercel CLI 发布，不依赖后端服务。
